# План тестирования (Test Plan)

**Версия:** 1.0  
**Объект:** консольный P2P-чат на Kotlin/JVM (`ExtremeProgrammingPP`).  
**Нормативные ссылки:** `docs/Требования.md`, `docs/Архитектура.md`, контракт `chat-api/.../chat.proto`.

---

## 1. Введение и цели

**Цель документа:** зафиксировать стратегию и объём тестирования, критерии завершения, связь требований с тестами и фактическое покрытие (чеклист).

**Цели тестирования:**

- подтвердить соответствие реализации функциональным и основным нефункциональным требованиям к чату, CLI и транспорту gRPC;
- снизить регрессии при изменениях в `:chat-api`, `grpc.*`, `cli.*`, `domain.*`;
- обеспечить повторяемую автоматическую проверку в CI.

## 2. Область (Scope)

**В объёме:** доменная валидация и команды; разбор аргументов и вывод CLI (с подменой потоков/транспорта); маппинг protobuf ↔ домен; интеграционные сценарии gRPC-транспорта на localhost; негативные сценарии транспорта (один peer, отказ второго клиента; разрыв); смоук запуска приложения; опционально — E2E два процесса через бинарь `installDist`.

**Вне объёма:** нагрузочное и стресс-тестирование; тестирование UI не-консольного; полная совместимость разных версий proto между независимыми релизами; тесты на уровне железа и ОС за пределами заявленных (Linux/Windows в CI); отдельный аудит безопасности TLS (в проекте используется plaintext для учебного сценария).

## 3. Стратегия и уровни тестирования

| Уровень | Назначение | Инструменты |
|--------|------------|-------------|
| **Модульный (unit)** | Изолированная логика домена, парсер CLI, рендерер без сети | Kotlin Test, JUnit Platform |
| **Интеграционный** | gRPC-контракт, `GrpcChatTransport`, отрицательные сценарии | То же + localhost |
| **Системный / E2E** | Два процесса приложения из дистрибутива, обмен сообщением | Отдельная Gradle-задача `e2eTest`, тег JUnit `e2e` |
| **Смоук** | Приложение стартует, CLI отвечает на `--help` | CI: `gradlew run --args="--help"` |

**Принцип:** критичная логика и контракт покрыты автотестами в основной задаче `:test`; E2E вынесен из `:test` (исключён тег `e2e`) из‑за зависимости от `installDist` и большей длительности.

## 4. Критерии входа и выхода

**Вход (для прогона полного набора):** успешная сборка проекта (`assemble` или эквивалент), JDK 21, доступ к сети localhost для интеграционных тестов.

**Выход — успешное завершение:**

- задача `./gradlew test` завершается без падений (E2E не входит);
- в CI: дополнительно `e2eTest` и смоук `run` с `--help` — по конвейеру в `.github/workflows/ci.yml`.

**Критерий провала:** любой упавший тест в обязательной цепочке CI для ветки `main` / `master`; блокирующий дефект — падение интеграционного теста транспорта или регресс в разборе CLI.

## 5. Среда тестирования

- **Язык / рантайм:** Kotlin JVM, toolchain **21** (см. `build.gradle.kts`).
- **Сборка:** Gradle Wrapper; модуль контракта `:chat-api` (protobuf / gRPC codegen).
- **CI:** GitHub Actions, матрица **ubuntu-latest**, **windows-latest**.
- **Локально:** WSL/Linux или Windows; для E2E и ручной проверки рекомендуется бинарь `installDist`, а не только `gradlew run` (интерактивный stdin).

## 6. Трассировка требований к тестам (сжатая матрица)

| Требование (суть) | Как проверяется |
|---------------------|-----------------|
| P2P, два участника, прямое соединение | `GrpcChatTransportIntegrationTest`, E2E `endtoend.CliTwoProcessE2ETest` |
| Режим сервера / клиента по аргументам | `CommandLineParserTest`, `ChatCliAppTest`, E2E |
| Имя, дата/время, текст в сообщении | `ConsoleRendererTest`, `ProtoMappingTest`, интеграция транспорта |
| Консольный ввод-вывод, команды `/help`, `/exit` | `ChatCliAppTest`, `ConsoleRendererTest` |
| gRPC + protobuf контракт | `EchoGrpcIntegrationTest`, `ProtoMappingTest`, транспортные тесты |
| Дуплекс после соединения | `GrpcChatTransportIntegrationTest` |
| Предсказуемость при разрыве | `GrpcChatTransportNegativeTest` (disconnect), события `Disconnected` |
| Один активный peer на сервере | `GrpcChatTransportNegativeTest` (второй клиент → `RESOURCE_EXHAUSTED`) |
| CI, автотесты | Раздел 7, workflow CI |

Полная детализация требований — в `docs/Требования.md`.

## 7. Автоматизация и задачи Gradle

| Задача | Содержание |
|--------|------------|
| `test` | Юнит + интеграционные тесты; тег **`e2e` исключён** |
| `e2eTest` | Зависит от **`installDist`**, только тег **`e2e`** (`endtoend.CliTwoProcessE2ETest`), свойство `e2e.binary` |
| CI | `assemble` → `test` → `e2eTest` → `run --args="--help"` (Linux и Windows) |

Стек: **Kotlin Test**, **JUnit Platform**.

## 8. Чеклист реализованных тестов (факт)

| Слой | Файлы | Суть |
|------|--------|------|
| Домен | `PeerInfoTest`, `ChatMessageTest`, `CommandTest` | Валидация, команды |
| CLI | `CommandLineParserTest`, `ConsoleRendererTest`, `ChatCliAppTest` | Парсинг, вывод, сценарии с фейковым транспортом |
| gRPC | `ProtoMappingTest`, `EchoGrpcIntegrationTest` | Маппинг, эхо по контракту |
| Транспорт | `GrpcChatTransportIntegrationTest`, `GrpcChatTransportNegativeTest` | Дуплекс; отказ второго клиента; disconnect |
| E2E | `endtoend.CliTwoProcessE2ETest` | Два процесса бинарника, обмен строкой |

Модуль `:chat-api` отдельных тестов не содержит (только генерируемый код и контракт).

## 9. Риски и запланированные дополнения

| Риск / пробел | Мера |
|-----------------|------|
| Нет изолированных тестов на `PeerAddressInterceptor` / `ServerCall` | Запланировано: моки gRPC `ServerCall`, проверка контекста peer |
| Ручная проверка двух терминалов не формализована как чеклист | По необходимости — по README / `scripts/demo_alice_bob.py` |
| E2E зависит от окружения и длительности | Отдельная задача `e2eTest`, не блокирует быстрый `test` |
