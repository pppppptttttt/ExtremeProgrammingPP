# Требования

## 1. Функциональные требования

### 1.1. Сетевая модель
- Приложение реализует **peer-to-peer чат**: два процесса соединяются **напрямую**, без обязательного центрального сервера сообщений.
- Один участник может работать в режиме **ожидания входящего подключения**.
- Второй участник может подключаться к нему по **адресу и порту**.

### 1.2. Режимы запуска
Приложение должно поддерживать два сценария запуска:

#### Серверный режим
- пользователь указывает имя;
- пользователь запускает приложение без адреса peer-а;
- приложение начинает слушать указанный локальный порт и ждёт входящего подключения.

#### Режим подключения к peer-у
- пользователь указывает имя;
- пользователь указывает адрес и порт удалённого peer-а;
- приложение инициирует исходящее подключение.

### 1.3. Обмен сообщениями
- После установления соединения оба участника могут:
  - отправлять сообщения;
  - принимать сообщения;
  - видеть сообщения собеседника в консоли.
- Сообщения передаются в дуплексном режиме.

### 1.4. Отображение сообщений
Для каждого сообщения в консоли должны отображаться минимум:
- имя отправителя;
- дата и время отправки;
- текст сообщения.

### 1.5. Пользовательский интерфейс
- Интерфейс приложения — **консольный**.
- Ввод сообщений выполняется через стандартный ввод.
- Вывод сообщений и служебных событий выполняется в консоль.

### 1.6. Сетевой контракт
- Обмен между peer-ами реализуется через **gRPC**.
- Формат сетевых сообщений задаётся в **Protocol Buffers**.
- Контракт хранится в отдельном модуле проекта: `chat-api`.

## 2. Нефункциональные требования

### 2.1. Технологии
- Язык реализации: **Kotlin/JVM**.
- Сетевой стек: **gRPC + Protocol Buffers**.
- Система сборки: **Gradle**.
- CI: **GitHub Actions**.

### 2.2. Архитектура
Проект должен быть разделён минимум на следующие логические части:
- **CLI**;
- **доменная модель**;
- **транспортный слой**;
- **protobuf/gRPC контракт** в отдельном модуле.

### 2.3. Поддерживаемость
- Код должен быть структурирован по пакетам и уровням ответственности.
- Публичные и нетривиальные части реализации должны быть документированы.
- Транспортный слой не должен напрямую протекать в CLI-логику без адаптера или порта.

### 2.4. Надёжность
- При ошибке подключения пользователь должен получить понятное сообщение.
- При разрыве соединения приложение должно завершаться корректно или явно уведомлять о состоянии.
- Завершение gRPC-канала и сервера должно выполняться корректно.

### 2.5. Тестирование
Должны быть предусмотрены:
- unit-тесты для критичной логики;
- integration-тесты для сетевого контракта и транспорта;
- автоматический запуск тестов в CI.

### 2.6. Кроссплатформенность
Проект должен собираться и проходить тесты минимум на:
- Linux;
- Windows.
