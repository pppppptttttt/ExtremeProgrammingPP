# Требования

## Функциональные

### Сетевая модель и запуск
- Система реализует **peer-to-peer**: два участника устанавливают соединение **напрямую** (без обязательного центрального сервера приложения).
- При запуске задаётся **имя пользователя** (отображается у собеседника как отправитель сообщений).
- Режимы запуска:
  - **Клиент / исходящее подключение**: указываются **адрес** peer-а и **порт** для подключения к уже слушающему участнику.
  - **Сервер / ожидание**: адрес и порт peer-а **не указываются** — процесс переходит в режим **ожидания входящего подключения** на заданном локальном порту (порт задаётся явно в конфигурации запуска или по соглашению, см. нефункциональные ограничения).

### Интерфейс и отображение
- **Консольный пользовательский интерфейс** (CLI): ввод текста в терминале, вывод входящих сообщений в тот же или связанный поток вывода.
- Для каждого сообщения отображаются минимум:
  - **имя отправителя**;
  - **дата и время отправки** (согласованная таймзона/формат фиксируются в реализации и документации);
  - **текст сообщения**.

### Протокол и разделение реализации
- Обмен данными между участниками осуществляется по **gRPC** поверх описанного в **Protocol Buffers** контракта (`.proto`).
- Репозиторий структурирован с **разделением ответственности** (например: отдельные модули/подпроекты для общего контракта, сетевого слоя, CLI и доменной логики чата), согласованное с командой **декомпозирование задач** фиксируется в архитектурном документе.

### Сопутствующее (из условий сдачи)
- В репозитории размещаются **формализованные требования** (настоящий документ), **архитектурное описание** с диаграммами компонентов и классов, **обоснование выбора технологий**, **распределение работ** между участниками.
- Отдельный документ: **план тестирования** и **чеклист** реализованных и запланированных тестов (полный набор тестов не обязателен, но **основной костяк** должен быть реализован).

## Поведенческие и правила обработки
- После установления соединения оба участника могут **отправлять и принимать** сообщения (дуплексный обмен в рамках выбранного gRPC API — streaming или unary по согласованному контракту).
- При разрыве соединения поведение должно быть **предсказуемым**: корректное завершение или понятное сообщение об ошибке, отсутствие «зависания» CLI без обратной связи (детали — в README и плане тестирования).
- Формат даты/времени и кодировка текста консоли согласованы между клиентами одной версии протокола; при несовместимости версий — явная ошибка или согласованная политика (описать в README).
- Имя пользователя задаётся **локально при запуске** и передаётся собеседнику в рамках протокола так, чтобы отображалось в списке/ленте сообщений.

## Нефункциональные
- **Язык реализации**: Kotlin (JVM), сборка и зависимости — через принятый в проекте инструмент (Gradle).
- **Контракт**: gRPC + Protocol Buffers; версионирование `.proto` при изменениях API.
- **Разделение на модули/слои**: чёткие границы между транспортом (gRPC), моделью сообщений чата и CLI; избегать монолитного «клубка» классов.
- **Документирование кода**: комментарии к публичным API, классам и нетривиальной логике (уровень согласуется с правилами курса и линтером).
- **Тестирование**: юнит- и/или интеграционные тесты для критичной логики и контракта; остальное — в плане тестирования.
- **Сборка и запуск**: одной или нескольким командами из **README**; оцениваемая функциональность — из ветки **main** (или **master**).
- **Индустриальные практики**: логирование значимых событий (подключение, ошибки, отключение), осмысленная обработка ошибок сети и ввода-вывода.

## Требования к оформлению репозитория и CI
- **README.md**: назначение проекта, как собрать и запустить (оба режима: «сервер» и «клиент»), примеры команд, кратко — устройство реализации.
- **.gitignore**: отсутствие артефактов сборки, локальных IDE-файлов и секретов в репозитории.
- **CI**: автоматическая **сборка**, по возможности **линтер** и **тесты**; осмысленные сообщения коммитов, описания PR, краткие ревью PR (по требованиям курса).
- Репозиторий не захламлять лишними файлами; при необходимости — лицензия и согласованность лицензий зависимостей.
- Ветка по умолчанию для проверки работы — **main/master**; инструкции в README должны соответствовать фактическому способу запуска.

## Область и допущения
- Полный перечень пользовательских сценариев и ограничений (лимиты длины сообщения, переподключение, несколько peer-ов) может уточняться по мере реализации; всё, что **не** реализовано в main, должно быть явно отмечено в README или в требованиях как запланированное/вне scope.
